Literature Vault — Documentation

The on-disk contract that the Viewer and the Librarian agent share, the algorithms that produce it, and the stack the viewer is built on. For installation, return to the landing page; for the full architectural design rationale, see docs/DESIGN.md in the repository.

01Vault layout

The vault is an ordinary directory. Both surfaces — the macOS Viewer and the Librarian agent — read and write the same files; there is no database and no API between them. Filename equals citekey for the three primary file types (.md, .bib, .pdf), so a paper's record can be reconstructed from any one of them. 

PaperNotes/{citekey}.md       literature note (YAML + body)
Bibfiles/{citekey}.bib        BibTeX, one per paper
PDFs/{citekey}.pdf            the actual paper
SI/{citekey}-SI.pdf           supplementary information
Annotations/{citekey}.json    viewer highlights (EmbedPDF sidecar)
Inbox/                        drop zone for new PDFs
Collections/{slug}/           user-curated paper groupings
Projects/                     curated topic notes (user-authored)
index.json                    dedup index (sha256, DOI, arXiv)
library.bib                   aggregated from Bibfiles/*
embeddings.db                 sqlite-vec, regenerable
.embedding_models/{name}/     model cache (gitignored, regenerable)

Who writes what

The Librarian writes every file type. The Viewer writes PaperNotes/, Collections/, Projects/, Annotations/, and Inbox/ (when accepting a drag-and-dropped PDF). The remaining files — Bibfiles/, index.json, library.bib, embeddings.db — are written only by the deterministic scripts the agent invokes, never edited by hand.

Concurrency

Every script that modifies a shared file (notes, index, aggregated bib) uses the tempfile-plus-rename pattern, so a crash mid-write cannot leave partial state in the vault. The Viewer follows the same convention for the files it owns.

02Citation keys

Citekeys are permanent. Once a paper has been filed under a citekey, the key never changes — every other piece of state (filename, bibliography entry, collection membership, embedding row) is keyed off it.

Algorithm

Given a DOI, the canonical citekey is:

key = "{author}{year}-{journal}-{doi_suffix}"
  • author — first-author surname, Unicode-normalised, lowercased, restricted to [a-z].
  • year — CrossRef published / published-print / issued, first non-null.
  • journal — short abbreviation from a curated map (prl, jcp, natcommun, …).
  • doi_suffix — DOI tail (after the first /), slugified, with any redundant {journal}- prefix stripped.

Fallbacks

Papers without a DOI use the arXiv form {author}{year}-arxiv-{arxiv-slug}. Books, theses, and other pre-digital references fall back to a manual form {author}{year}-{short-title-slug} minted by scripts/manual_file.py.

Examples

DOI / source Citekey
10.1103/PhysRevLett.126.236001 cheng2021-prl-physrevlett-126-236001
10.1021/acs.jpclett.4c00605 stolte2024-jpcl-4c00605
10.1073/pnas.2501728122 juel2025-pnas-2501728122
10.1038/s41467-023-36329-y musaelian2023-natcommun-s41467-023-36329-y
arXiv:2305.10141 musaelian2023-arxiv-2305-10141

Reference implementation: scripts/doi2bib.py:make_canonical_key. Full grammar: docs/CITATION_KEYS.md.

03Note frontmatter

Every PaperNotes/{citekey}.md begins with a YAML frontmatter block. The required fields are citekey, title, authors, year, journal, doi, arxiv_id, added, and sha256_pdf; an abstract block is included when available. Additional fields may be added freely. 

---
citekey: tromborg2011-prl-physrevlett-107-074301
title: "Transition from static to kinetic friction…"
authors:
  - "Trømborg, Jørgen"
  - "Scheibert, Julien"
  - "Amundsen, David Skålid"
  - "Thøgersen, Kjetil"
  - "Malthe-Sørenssen, Anders"
year: 2011
journal: "Physical Review Letters"
doi: 10.1103/PhysRevLett.107.074301
arxiv_id: null
added: 2026-04-22
tags: ["friction", "rate-and-state"]
sha256_pdf: 7e3d9a…
abstract: |
  We investigate the transition…
---

Body sections

The body below the frontmatter is free-form Markdown. Three section headings are conventional and recognised by the scripts and the agent:

  • ## Why — why the paper is in the library; short.
  • ## Cleaned Notes — synthesised summary, rewritten by the agent when raw notes change.
  • ## Notes — timestamped log appended by append_note.py (header format ### {ISO timestamp}).

The Viewer preserves everything outside the frontmatter block byte-for-byte, so co-editing the same notes from Obsidian or a terminal editor is safe.

Tag conventions

Topic tags are lowercase snake_case (friction, ice_friction, mlip). Citation-prep tags use the cite: prefix (cite:thesis-3, cite:helfo-rebuttal) — one per manuscript-in-progress, removed when the manuscript is submitted. The Viewer surfaces cite:* tags as a separate sidebar section so a manuscript becomes a live view of its sources.

04Collections

Collections are user-curated groupings of papers. They are distinct from frontmatter tags: — a paper does not get tagged thesis-3 to belong to a thesis-3 collection; instead its citekey is listed in that collection's index.md. Membership is always explicit; sub-collections nest.

Directory format

Every collection is a directory under Collections/ with an index.md inside. The directory name is the collection's slug; the relative path from Collections/ is the collection's full identifier (e.g. drafts/thesis-3). 

Collections/drafts/thesis-3/index.md

---
name: thesis-3
slug: drafts/thesis-3
description: Confined water in clay layers …
created: 2026-04-12T11:02:00+02:00
updated: 2026-05-07T15:20:00+02:00
---

# Confined water in clay layers
<free-form prose, never rewritten>

## Papers
- marry2018-jpcc-…
- guren2021-geca-… — table 3 has the diffusion data
- claverie2022-cacr-…

The ## Papers section is the only part of the file the scripts and agent write to. Bullets are plain citekeys (- {citekey}); an optional inline annotation may follow an em-dash. Anything before ## Papers (the H1, free-form prose) and anything after it is preserved byte-for-byte.

Operations

  • Create: mkdir -p Collections/{slug}, write index.md with frontmatter and an empty ## Papers section. Set created and updated to the current ISO datetime with offset.
  • Add a paper: append - {citekey} to ## Papers if not already present; bump updated.
  • Remove a paper: delete the matching bullet; bump updated.
  • Rename / move: mv Collections/{old} Collections/{new} and rewrite the slug: and name: fields in the moved index.md.

All writes go through a tempfile in the same directory followed by os.replace.

05Intake pipeline

Filing a new PDF is a deterministic, atomic transaction implemented in scripts/file_paper.py. The same script is invoked by the Librarian (Telegram intake) and by the Viewer's manual-entry dialog (offline intake).

  1. Compute the SHA-256 of the PDF.
  2. Extract DOI / arXiv ID from the PDF: metadata → text → raw bytes (three-tier fallback in extract_ids.py).
  3. If no identifier was found: search CrossRef by title and author (crossref_search.py); the operator confirms a match.
  4. Fetch CrossRef metadata; mint the canonical citekey; build the BibTeX entry.
  5. Duplicate-check against index.json by SHA-256, DOI, arXiv ID, and citekey.
  6. Atomic transaction: write Bibfiles/{citekey}.bib and a PaperNotes/{citekey}.md skeleton; move the PDF to PDFs/{citekey}.pdf; update index.json; regenerate library.bib.
  7. Roll back partial writes on any failure.

The skeleton note inherits abstract from CrossRef when present, or falls back to PDF-page-0 text. The Librarian audits this abstract field on every new filing — see agent/CLAUDE.local.md § When a PDF arrives for the full procedure.

06Embedding server

A small HTTP server (scripts/embed_server.py) runs on the host at 127.0.0.1:5817 and stores embeddings in embeddings.db. It is multi-model: the registry lives in scripts/embed_models.json, each model is lazy-loaded on first request, and a per-model idle TTL unloads cold models while keeping hot ones warm. The whole process exits after 30 min of full idle and is respawned by launchd on the next request.

Endpoints

  • GET /health — returns {ok, default, registered, loaded, uptime_sec, idle_sec}.
  • POST /embed — body {texts: [str], prompt: "document"|"query", model?: str}; returns {vectors, model}.
  • POST /shutdown — graceful exit.

Storage

The schema in embeddings.db (sqlite-vec) is:

  • papers (citekey, model, content_hash, embedded_at) — primary key (citekey, model).
  • vec_papers__<safe_model_id> (citekey TEXT PRIMARY KEY, embedding FLOAT[dim]) — one virtual table per registered model.

The corpus content embedded per paper is Title: {title}\n\nAbstract: {abstract}\n\nWhy: {why}\n\nNotes: {cleaned_notes}. embed_corpus.py re-embeds incrementally when the content hash changes.

07Viewer architecture

The Viewer is a Tauri desktop app — a Rust backend handling all filesystem I/O, with a Svelte frontend in the WebView. The frontend never touches files directly; it calls typed Rust wrappers over Tauri's IPC.

Frontend

Svelte 5 with runes ($state, $derived, $effect), Vite, TypeScript. No external state library; per-feature stores in .svelte.ts modules.

Backend

Tauri 2 + Rust. All vault reads and writes live behind Rust commands; atomic-write helpers are in src-tauri/src/atomic.rs. Settings (the active vault path) are persisted to ~/Library/Application Support/literature-vault/settings.json via src-tauri/src/settings.rs.

PDF

@embedpdf/* with the PDFium engine for rendering, plus a custom Svelte overlay for highlights. Annotations are persisted as EmbedPDF AnnotationTransferItem[] JSON sidecars at Annotations/{citekey}.json, filtered against a baseline snapshot so PDFs that arrive with embedded annotations don't pollute the sidecar.

Editor

CodeMirror 6 (Markdown mode, line wrapping, history). Notes are byte-preserving — only the YAML block and body bytes change on save.
