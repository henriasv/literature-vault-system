Literature Vault: a reference manager system for the BibTeX + Markdown workflow

Henrik Sveinsson1*

Literature Vault is a small, opinionated reference manager system for researchers who write in LaTeX, Typst, or Pandoc. Two surfaces share one folder on disk: the Viewer on macOS for deep reading and note-writing, and the Librarian — a Telegram bot running on nanoclaw — for phone-based intake and conversational library follow-up. PDFs on disk, one Markdown note per paper, BibTeX out; the contract between surfaces is the filesystem, and nothing else. The vault is independent of either tool.

The Viewer · macOS, Tauri 2 + Svelte 5 · The Librarian · Telegram, via nanoclaw

Who it's for

Researchers who write in LaTeX, Typst, or Pandoc and want their reading material as plain files on disk. If your paper-writing flow already starts with \cite{…}, the system is aimed at this workflow — opinionated about format, deliberately small in scope, and structured so the files are independent of either tool.

Two surfaces, one vault

The Viewer (macOS app) is for deep reading and note-writing — library list, PDF, Markdown notes side-by-side, semantic search.
The Librarian (Telegram bot, via nanoclaw) is for phone-based intake and chat-based library follow-up.
They share one folder on disk and nothing else. Swap either piece without breaking the other.

No. 003 · Install

Six commands, in order.

Apple Silicon Mac. The first script that takes --vault stores it in a gitignored setup/.local.conf; the rest default to it. Steps 5 and 6 are optional.

  1. Prerequisites — Xcode CLI tools, Homebrew, Rust, Node, uv.

    xcode-select --install
brew install rustup-init node uv
rustup-init -y

  2. Clone the repo and build the Viewer.

    git clone https://github.com/henriasv/literature-vault-system
cd literature-vault-system

./setup/install-viewer.sh

    Installs Literature Vault.app into /Applications/. The Rust compile takes a few minutes the first time.

  3. Scaffold a vault.

    ./setup/init-vault.sh ~/literature-vault

# already have papers + notes? add --adopt:
./setup/init-vault.sh ~/existing-collection --adopt

  4. Point the Viewer at it.

    ./setup/configure-viewer.sh

  5. Embed server — optional
    Always-on semantic search via launchd.

    Default model is nvidia/llama-embed-nemotron-8b (4096-dim, ~14 GB). Run from inside the vault so the path resolves correctly. 

    cd ~/literature-vault

# Authenticate (the Nemotron repo is gated):
uv run --with huggingface_hub huggingface-cli login

# Download into .embedding_models/<dir-name-in-embed_models.json>/:
uv run --with huggingface_hub python3 -c '
from huggingface_hub import snapshot_download
snapshot_download(
    repo_id="nvidia/llama-embed-nemotron-8b",
    local_dir=".embedding_models/llama-embed-nemotron-8b")'

# Index the corpus + launch the server:
uv run scripts/embed_corpus.py
./setup/install-embed-server.sh

    Lighter alternative: google/embeddinggemma-300m (768-dim, ~1.2 GB) — flip default in scripts/embed_models.json if you install it instead.

  6. Librarian agent — optional
    Telegram intake via nanoclaw.

    First, on the nanoclaw side:

    1. Clone and install nanoclaw per its own README (Docker + an Anthropic API key are the main prerequisites).
    2. Run nanoclaw's onboarding flow to pair a Telegram bot. You'll need a bot token from @BotFather; nanoclaw walks you through the pairing.
    3. Confirm nanoclaw is healthy — DM the bot once and verify you get a reply.

    Then, back in this repo: 

    ./setup/install-agent.sh --nanoclaw ~/repos/nanoclaw

    This creates a new group at ~/repos/nanoclaw/groups/librarian/ with a rendered container.json (vault + system mounts) and the agent's CLAUDE.local.md; it also patches nanoclaw's mount allowlist. Start the group through nanoclaw's usual workflow.

    First run inside the container bootstraps uv + Python into vault/.bin/ — ~30 s, looks like a hang but isn't.

No. 004 · The Viewer · on your Mac

Built for reading and noting.

Drop & file

Drag in a PDF; the viewer files it.

Drop PDFs on the window (or ⌘N) and the Viewer files them directly — DOI extracted, citekey minted, note skeleton written, library.bib updated. No external agent in the loop.

Three-pane reader

Library, PDF, notes — one window.

Library list on the left, PDF in the middle, Markdown note on the right. Open multiple papers in tabs. Page, zoom, and scroll position persist per tab across launches.

Markdown notes

Rendered article or raw editor.

Toggle between a typeset rendered view and CodeMirror 6. ⌘⇧N appends a timestamped section so a reading session leaves a dated trail in the note.

Semantic search

Query the whole collection by meaning.

Flip the SEM switch in the search bar to query embeddings.db through your embed server. Falls back quietly when the server is unreachable.

Tags & manuscripts

Manuscripts as a live view of their sources.

Per-paper tag pills with autocomplete from the whole library. cite:* tags surface as a Manuscripts sidebar section, so a manuscript becomes a live view of its sources.

No. 005 · Plays well with Obsidian

Plain Markdown, end to end.

Every paper's note is a plain .md file with a fixed YAML frontmatter — citekey, title, authors, doi, tags, an abstract block, and your body text. Open the same folder in Obsidian and everything works: backlinks, graph view, tag panes. The viewer reads what's on disk; no lock-in.

The viewer preserves byte-for-byte everything outside the YAML block and body, so co-editing the same notes from Obsidian or your terminal editor is safe. 

~/literature-vault/
├── PaperNotes/{citekey}.md     # one note per paper
├── PDFs/{citekey}.pdf          # the actual papers
├── Bibfiles/{citekey}.bib      # bibtex (agent-owned)
├── SI/{citekey}-SI.pdf         # supplementary info
├── Annotations/{citekey}.json  # viewer highlights (viewer-owned)
├── Inbox/                      # drop zone
├── index.json                  # dedup (agent-owned)
├── library.bib                 # combined bib (agent-owned)
└── embeddings.db               # sqlite-vec (agent-owned)

No. 006 · The Librarian · deep dive

Intake, follow-up, and chat queries.

Phone intake

Send a PDF or arXiv link from anywhere; metadata extracted, citekey minted, BibTeX written, dedup checked, note scaffolded.

Reading context as you go

Reply to the bot with why a paper made it into your library — what your colleague said about it, a figure you want to remember, how it connects to something you're reading. It gets appended to the note's ## Why or ## Notes section.

Library queries by chat

"My papers on rate-and-state friction" → find_similar.py over embeddings.db → citekeys back, ranked. Useful when you're away from your laptop.

Scheduled jobs

Weekly reading digests, dedup audits, embedding refresh — recurring tasks run by the nanoclaw container. Each agent group has its own CLAUDE.local.md with your conventions.

The Librarian runs as a nanoclaw agent bound to a Telegram channel (Discord, Slack, iMessage, WhatsApp, … all supported by nanoclaw if you'd rather not use Telegram). The contract between Viewer and Librarian is the filesystem, not an API; any agent that produces files in the vault layout will work.
